How to read this doc

This document is the design counterpart to PLAN.md. PLAN.md is the architectural source of truth (tech stack, schemas, agent loop, deployment). This doc is the visual & interaction source of truth (tokens, components, layouts, motion, states). Where the two overlap, this doc illustrates what PLAN.md specifies — they should never disagree; if they do, raise it.

The interactive prototype that informed every spec below lives at Hyble AI Menu MVP.html in the design project. Open it side-by-side with this doc — for any "how should this feel" question, the prototype is the answer.

Build order Phases 1–5 are defined in PLAN.md §10. This doc is structured to be read top-to-bottom once, then dipped into per-phase. Sections 02–05 are reference material (build the design system once); 06–14 are per-screen / per-state.
01

Visual direction

Editorial typographic, AI-as-engine

The app is a calm, magazine-grade workspace where the generated menu is the only loud thing on screen. The chrome is restrained — wine-industry sophistication without enterprise stiffness, AI capabilities without consumer-toy sparkle. We selected this direction (over a chat-playground or Canva-clutter alternative) because reps are time-poor, often working in dim bars, and a cluttered chrome competes with the menu they need to see.

Principles

  1. The menu is the hero. App chrome never uses customer brand colors; those only appear inside the generated artwork, on customer dropdown badges, and on Orders metadata pills.
  2. One accent. Oxblood in light, warmer rust in dark. Used only on primary CTAs, active rail state, focus rings, and inline-link emphasis. Anything else is ink/paper.
  3. AI is invisible. No gradients on chrome, no shimmer, no sparkle icons. The streaming SSE skeleton (see §08) is the AI cue — and that's enough.
  4. Copy is warm and human. "Let's build a menu," "We'll save your draft," "Thanks for the photo." Industry-savvy without slang.
  5. Print-grade typography. Display serif for the prompt sentence, screen titles, and marketing surfaces only. Mono for metadata, eyebrows, status. Sans for everything else.

Anti-references

Avoid: sparkle/star iconography, gradient hero panels, glassy translucent surfaces, Canva-style toolbar density, ChatGPT's bare playground UI, generic SaaS card-grid dashboards, soft-rounded 24px+ corners, mascots / illustrations.

Switching

System default on first load (prefers-color-scheme); manual override persisted in localStorage["hyble.theme"] as "light" | "dark" | "system". The toggle lives in the rail footer (desktop) and in Settings (mobile, post-MVP). All components must use the CSS variable tokens — no hardcoded colors anywhere — so theme swap is instant and complete.

Contrast

All --ink/--paper pairs and --accent/--accent-ink pairs verified against WCAG AA (≥ 4.5:1 for body, ≥ 3:1 for large/UI). --ink-3 on --paper is borderline AA Large only — use it for secondary meta only, never for required information.

Theme switch hook

// apps/web/src/lib/theme.ts
export type Theme = "light" | "dark" | "system";
export function applyTheme(t: Theme) {
  const resolved =
    t === "system"
      ? (matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light")
      : t;
  document.documentElement.dataset.theme = resolved;
  localStorage.setItem("hyble.theme", t);
}

07

Create flow — initial → active

Initial — the prompt sentence

The most important component in the product. A single sentence the user fills in, set in the display serif at 32–40px depending on viewport. Slots are inline-block elements with a dashed underline that opens a popover on click.

         "Let's build a menu."
         (eyebrow · ink-3 · mono)

         Create a [type ▾] for [customer ▾]
         with this menu [+ image] and make
         the following changes: [free text…]

                            [ Generate → ]

Slots:

  • type — popover with two large radio cards: Wine menu, Cocktail menu.
  • customer — Combobox. Selecting pulls in brand kit + primary state. Selected state shows brand color dot + name + state badge.
  • image — optional upload. Desktop: drag-drop on slot popover + file input. Mobile: native sheet, capture=environment.
  • free text — auto-grow textarea inline. Empty state shows ghost copy.

Generate button disables until type and customer are set. Image and free-text are optional. The presence of image flips the agent into recreate mode (~60% of orders per PLAN.md §2).

Active — split layout

Each new chat turn triggers a new generation. The right panel always shows the latest generation; clicking a thumbnail in chat history restores that generation to the right panel non-destructively (does not branch the order — parent_generation_id still chains the next turn from the latest, per PLAN.md §4).

SSE generating state

SSE events from PLAN.md §3 drive both the chat-side checklist and the design-side reveal. The menu skeleton renders at the correct dimensions from the start so the page never reflows when the design lands.

event
visual
reading_reference
Top rule + customer brand band materialise on the menu surface. Chat: "Reading the menu you sent…" with spinner.
building_prompt
Headline + subheadline reveal. Chat: "Composing the prompt."
generating
Body items fill in row-by-row over ~2.2s. Reveal value 0.05 → 0.95. Chat: "Generating."
validating
Footer disclaimer materialises. Chat: "Validating compliance."
extracting_metadata
Last 5%. Generation thumbnail chip appears in chat history. Chat: "Tagging metadata."
done
Menu hands off to the live render. Status pill turns ok-green for 600ms, then fades.
error
Skeleton freezes. Inline alert in chat thread (see §09).

Implementation

// apps/web/src/lib/sse.ts
export function streamGenerate(orderId: string, onEvent: (e: SseEvent) => void) {
  const es = new EventSource(`/api/orders/${orderId}/generate/stream`);
  ["reading_reference","building_prompt","generating","validating","extracting_metadata","done","error"]
    .forEach(name => es.addEventListener(name, ev => onEvent({ name, data: JSON.parse(ev.data) })));
  return () => es.close();
}

Status copy is the only place AI vocabulary peeks through. Lock these to active verbs in present continuous and align them with whatever the agent loop ends up calling its phases.

09

States — error, empty, offline

Error (3 flavors, all inline in chat thread)

CauseTreatmentRecovery
API failure (5xx)Soft Alert. ink-2 body, neutral border."Try again" button. Cost not charged.
Compliance blockAlert with accent-tinted left rule."Apply fix and retry" — auto-applies the missing disclaimer/edit and re-runs.
Rate limit / capAlert with cap counter (32/50 today)."Save draft" action. Marketing Ops users see admin-lift link.

Empty

  • Orders empty — paper-textured menu placeholder with three rule lines, "No orders yet." in display serif (28px), single accent CTA "Create your first menu".
  • Customer dropdown empty — "No customers match that search" with link "Request a customer" (out of MVP scope; placeholder).

Offline

Service worker serves cached shell + last 5 orders + the latest generation per order (LRU). Compose-new is disabled; the prompt area shows a paper-card with copy "Offline — generations resume when you reconnect" and a passive online-detector that re-enables itself within 1s of navigator.onLine flipping.

12

Animation & motion

The morph — initial → active

The brief calls this out as the critical moment. The original prompt sentence becomes the first chat message; the page never feels like it navigated.

0–60ms
Button latch. Generate label fades to "…" and the button locks (aria-busy=true). No spinner — copy alone signals work.
60–360ms
The morph. The prompt sentence translates from page-center to the top of where the chat panel will live, scaling 1.0 → 0.7. transform-origin: top left. --ease-out. Concurrently the right half of the viewport reveals the design surface from a 24px translateX, opacity 0 → 1.
360–420ms
Split commits. Chat panel (38% / min 380px) and design surface separate via a 1px rule. The morphed sentence settles into the first chat-message card.
420ms+
SSE begins. "Hyble · working" header appears with the per-step checklist. Skeleton starts filling on the right (§08).

The reverse (active → initial) is not exposed. New menus start a new order from the rail.

Other motion rules

  • Popovers: 140ms scale 0.98 → 1, opacity 0 → 1, --ease-spring.
  • Toasts: slide-in 220ms from bottom-right; auto-dismiss 4s.
  • Theme switch: instant. No transition on color (it looks lazy at scale).
  • prefers-reduced-motion: morph collapses to opacity cross-fade only; no transform, no scale. Skeleton fill becomes a single fade-in.

Accessibility

  • All interactive surfaces meet WCAG AA contrast in both themes (verified §02).
  • Focus ring uses --accent, 2px outline + 2px offset. Never removed.
  • Slot popovers are real <button>s with aria-haspopup="dialog" and visible labels for screen readers — the visual "fill in a sentence" treatment is purely decorative on top.
  • Generating state announces SSE step transitions via aria-live="polite" on a hidden region.
  • Theme toggle is a real toggle button, not a checkbox; announces "Light theme active / Dark theme active".
  • Skeletons are role="status" with aria-label="Generating menu".
  • Keyboard shortcuts documented in a help dialog (? opens it; out-of-MVP nice-to-have).

16

Open questions & pushback

  1. Hyble brand kit availability. The system uses Hyble's name and v2 framing only — original visual identity. If there's an existing brand to honor (logo, color, type), grafting it in only requires changes to tokens.css and the wordmark.
  2. Customer count. The Combobox assumes 50–500 customers per rep. If it's tens of thousands, we need server-side typeahead and a "recent" cache, not a Combobox.
  3. "Save to order" semantics. PLAN.md treats orders and generations as separate; this design assumes every Generate creates a draft order on first run, and Save to order finalises it (sets final_generation_id). Confirm — or rework so generations are orphaned until explicitly saved.
  4. Cost visibility. Showing ~$0.04 next to Regenerate may feel hostile to reps. Recommend hiding from rep-tier users; show only to Marketing Ops.
  5. Image-only output for v1. Without img.ly, reps cannot fix a single typo without a full regeneration. At scale this will bite. Worth flagging to product.
  6. SSE step copy. Lock these to active verbs in present continuous; align names with whatever the agent loop ends up calling its phases.
  7. Mobile camera flow. Brief flagged dim-bar photo capture. capture=environment + native sheet handles this. Real-world testing essential — glare on glass-laminated menus is a known pain. Plan a 1-day on-site test in Phase 5.
  8. Reference-image cost. PLAN.md §6 flags this; UI surfaces per-call cost. Recommend also surfacing daily total in the rail footer for reps approaching their cap.
  9. Compliance UI for Marketing Ops. Out-of-scope, but somebody needs to maintain disclaimers and brand kits. Phase 6+ tee-up.
  10. Architectural pushback (PLAN.md §3). Single Fastify service serving the PWA + API is correct for MVP. I'd push back gently on "split into two later" — once chargeback Sprint 2 schemas land in shared/, the overhead of pnpm workspaces is the right time to also extract a serverless edge for the SSE endpoint. Streaming under load on a single Node process is the most likely first scaling pinch.
  11. PostHog masking (PLAN.md §11.9). Prompt textarea, chat composer, image-upload filenames, and customer dropdown values all carry potential PII. Add data-ph-no-capture to all four — flag for engineering at instrumentation time.
